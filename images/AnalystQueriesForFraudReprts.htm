Interaction: Fraud Report Results

1. Weak Access Control for a Resource  [State: Mitigation Implemented]  [Priority: High] 

Category:Information Disclosure
Description:Improper data protection of Fraud Transaction Repository can allow an attacker to read information not intended for disclosure. Review authorization settings.
Justification:Using X-Pack Security Role based access mechanism enables to authorize users by assigning privileges to users.

2. Spoofing of Source Data Store Fraud Transaction Repository  [State: Mitigation Implemented]  [Priority: High] 

Category:Spoofing
Description:Fraud Transaction Repository may be spoofed by an attacker and this may lead to incorrect data delivered to 1.0 Fetch Reported Fraud Activities. Consider using a standard authentication mechanism to identify the source data store.
Justification:Elasticsearch Sheild[2.4] helps in authenticating users or integrating with external user management.

Interaction: JSON Response

3. Spoofing of the Data Analyst External Destination Entity  [State: Mitigation Implemented]  [Priority: High] 

Category:Spoofing
Description:Data Analyst may be spoofed by an attacker and this may lead to data being sent to the attacker's target instead of Data Analyst. Consider using a standard authentication mechanism to identify the external entity.
Justification:Spoofing can be prevented by configuring the settings and enabling ssl.verification_mode. Elasticsearch provides these features in X-Pack Security.

4. External Entity Data Analyst Potentially Denies Receiving Data  [State: Mitigation Implemented]  [Priority: High] 

Category:Repudiation
Description:Data Analyst claims that it did not receive data from a process on the other side of the trust boundary. Consider using logging or auditing to record the source, time, and summary of the received data.
Justification:To prevent users from denying actions elasticsearch uses audit logging. This helps as a proof to prevent non-repudiation.

5. Data Flow HTTPS Is Potentially Interrupted  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:An external agent interrupts data flowing across a trust boundary in either direction.
Justification:Elasticsearch audit logs all the events and can be configured to alert on suspicious events. These type of attacks will use a similar URL patterns with changing parameters and using kibana we can integrate these audit logs with others to monitor suspicous requests trying to access the web server.

Interaction: Query DSL

6. Potential Excessive Resource Consumption for 1.0 Fetch Reported Fraud Activities or Fraud Transaction Repository  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:Does 1.0 Fetch Reported Fraud Activities or Fraud Transaction Repository take explicit steps to control resource consumption? Resource consumption attacks can be hard to deal with, and there are times that it makes sense to let the OS do the job. Be careful that your resource requests don't deadlock, and that they do timeout.
Justification:Settings can be made to create an alert that would notify you once a certain number of “404s” are triggered over a specific period of time. This can configured using Elastic Stack.

13. Fetch Reported Fraud Activities May be Subject to Elevation of Privilege Using Remote Code Execution  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation Of Privilege
Description:Data Analyst may be able to remotely execute code for 1.0 Fetch Reported Fraud Activities.

Category:Elevation Of Privilege
Description:An attacker may pass data into 1.0 Fetch Reported Fraud Activities in order to change the flow of program execution within 1.0 Fetch Reported Fraud Activities to the attacker's choosing.
Justification:Elasticsearch uses encrypted token patterns to prevent an attacker from making false data modifications or requests.
