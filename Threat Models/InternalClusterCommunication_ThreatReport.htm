1. Spoofing of Destination Data Store Data Index  [State: Mitigation Implemented]  [Priority: High] 

Category:Spoofing
Description:Data Index may be spoofed by an attacker and this may lead to data being written to the attacker's target instead of Data Index. Consider using a standard authentication mechanism to identify the destination data store.
Justification:The data indexes are stored in the file system and the path at which the indexes are stored is configured in a configuration file. If the configuration file has proper file permissions then the risk of spoofing should be relatively low.

2. Potential Excessive Resource Consumption for 1.0 Elasticsearch Document Storage or Data Index  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:Does 1.0 Document Indexing and Searching or Data Index take explicit steps to control resource consumption? Resource consumption attacks can be hard to deal with, and there are times that it makes sense to let the OS do the job. Be careful that your resource requests don't deadlock, and that they do timeout.
Justification:Since the data indexes are stored on the file system, a denial of service threat could be present if the Elasticsearch system receives data faster than it can write to disk. Generally, this will be more of a capacity management problem rather than an intentional threat, however attackers could attempt to flood several cluster nodes with new data in an effort to replicate data faster than that cluster can handle. Quotas on resource consumption could be an option to reduce this risk. A quality of service type system could also be beneficial if its desired to give a higher priority to writing new data locally rather than replicating data to other nodes.

4. Spoofing the Elasticsearch Cluster Node External Entity  [State: Mitigation Implemented]  [Priority: High] 

Category:Spoofing
Description:Elasticsearch Cluster Node may be spoofed by an attacker and this may lead to unauthorized access to 1.0 Document Indexing and Searching. Consider using a standard authentication mechanism to identify the external entity.
Justification:This is a high-risk threat since it crosses a threat boundary and an easy way for an attacker to attempt to steal data. Authentication is pretty much mandatory here. Elasticsearch OSS Threat Analysis: Elasticsearch performs authentication of cluster nodes by issuing and validating certificates for authorized hosts.

6. Potential Lack of Input Validation for 1.0 Elasticsearch Document Storage  [State: Mitigation Implemented]  [Priority: High] 

Category:Tampering
Description:Data flowing across non-replicated documents may be tampered with by an attacker. This may lead to a denial of service attack against 1.0 Document Indexing and Searching or an elevation of privilege attack against 1.0 Document Indexing and Searching or an information disclosure by 1.0 Document Indexing and Searching. Failure to verify that input is as expected is a root cause of a very large number of exploitable issues. Consider all paths and the way they handle data. Verify that all input is verified for correctness using an approved list input validation approach.
Justification:An attacker would in many cases need to circumvent several other controls like server authentication, network access control, and TLS encryption in order to tamper with the data flow, but input validation is still another needed control to verify that API requests are in the proper format and contain only data that meets the protocol specification. Each data replication request coming into the process should be checked to make sure they are structurally valid and conform to strict rules regarding expected input.

7. Potential Data Repudiation by 1.0 Elasticsearch Document Storage  [State: Mitigation Implemented]  [Priority: High] 

Category:Repudiation
Description:1.0 Document Indexing and Searching claims that it did not receive data from a source outside the trust boundary. Consider using logging or auditing to record the source, time, and summary of the received data.
Justification:There could many factors, including non-malicious factors that would require error checking and retransmitting a data replication request. In most cases it is probably not a significant risk to retransmit this data for replication purposes. A log that audits transactions between cluster nodes would be beneficial as well as a robust mechanism to keep track of which data is replicated on which nodes. Elasticsearch OSS Threat Analysis: Elasticsearch does offer an audit logging mechanism that includes some level of configuration to the destination of the logs. It also seems to have a complex process to write data to a primary node and then push data to one or more backup nodes.

8. Data Flow Sniffing  [State: Mitigation Implemented]  [Priority: High] 

Category:Information Disclosure
Description:Data flowing across non-replicated documents may be sniffed by an attacker. Depending on what type of data an attacker can read, it may be used to attack other parts of the system or simply be a disclosure of information leading to compliance violations. Consider encrypting the data flow.
Justification:This is a high-risk threat since it crosses a threat boundary and an easy way for an attacker to attempt to steal data. Except in the most controlled/restricted environments, or if processing completely non-sensitive data, encryption of data between cluster nodes is critical. Elasticsearch OSS Threat Analysis: Elasticsearch provides encryption between nodes via TLS.

9. Potential Process Crash or Stop for 1.0 Elasticsearch Document Storage  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:1.0 Document Indexing and Searching crashes, halts, stops or runs slowly; in all cases violating an availability metric.
Justification:Most denial of service risk across this threat boundary would be due to unwanted network traffic that could disrupt the internal cluster communication or overwhelm the host resources that the process depends on. Elasticsearch OSS Threat Analysis: Internal cluster communication can be separated from other traffic on a dedicated network interface on the host. Whitelisting is also available in Elasticsearch further restricting network access and reducing denial of service risk from network traffic.

10. Data Flow replicated documents Is Potentially Interrupted  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:An external agent interrupts data flowing across a trust boundary in either direction.
Justification:This would be hard for the application to control. Best practices for network security and network access control will be beneficial. The application could also make sure there are acknowledgements on any data replication requests so that at least the process knows when the replication was not successful.

11. 1.0 Elasticsearch Document Storage May be Subject to Elevation of Privilege Using Remote Code Execution  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation Of Privilege
Description:Elasticsearch Cluster Node may be able to remotely execute code for 1.0 Document Indexing and Searching.
Justification:Taking advantage of least privilege is a good tactic here. Running software as root or with more permissions than it needs will lead to even greater risk. Also restricting sensitive API calls to only specific roles or not making them accessible to the API at all could lower the risk. Elasticsearch OSS Threat Analysis: When installed from a default package the Elasticsearch software does not run as root.

12. Elevation by Changing the Execution Flow in 1.0 Elasticsearch Document Storage  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation Of Privilege
Description:An attacker may pass data into 1.0 Document Indexing and Searching in order to change the flow of program execution within 1.0 Document Indexing and Searching to the attacker's choosing.
Justification:If a remote cluster node could request tasks through the Elasticsearch API that are related to more than just data replication, then an attacker that gains control of one cluster node could use that to change the configuration of other nodes. Cluster nodes should have strict roles that only allow for data replication tasks.

13. Cross Site Request Forgery  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation Of Privilege
Description:Cross-site request forgery (CSRF or XSRF) is a type of attack in which an attacker forces a user's browser to make a forged request to a vulnerable site by exploiting an existing trust relationship between the browser and the vulnerable web site. In a simple scenario, a user is logged in to web site A using a cookie as a credential. The other browses to web site B. Web site B returns a page with a hidden form that posts to web site A. Since the browser will carry the user's cookie to web site A, web site B now can take any action on web site A, for example, adding an admin to an account. The attack can be used to exploit any requests that the browser automatically authenticates, e.g. by session cookie, integrated authentication, IP whitelisting, … The attack can be carried out in many ways such as by luring the victim to a site under control of the attacker, getting the user to click a link in a phishing email, or hacking a reputable web site that the victim will visit. The issue can only be resolved on the server side by requiring that all authenticated state-changing requests include an additional piece of secret payload (canary or CSRF token) which is known only to the legitimate web site and the browser and which is protected in transit through SSL/TLS. See the Forgery Protection property on the flow stencil for a list of mitigations.
Justification:Since Elasticsearch uses a single API for all inputs/outputs its probably unlikely that CSRF would be a significant risk.

17. Spoofing of Source Data Store Data Index  [State: Mitigation Implemented]  [Priority: High] 

Category:Spoofing
Description:Data Index may be spoofed by an attacker and this may lead to incorrect data delivered to 1.0 Document Indexing and Searching. Consider using a standard authentication mechanism to identify the source data store.
Justification:An attacker could potentially get unauthorized data replicated to other nodes in the cluster if the data storage path could be updated to a location that already contains the unauthorized data. If the configuration file has proper file permissions then the risk of spoofing should be relatively low.

18. Weak Access Control for a Resource  [State: Mitigation Implemented]  [Priority: High] 

Category:Information Disclosure
Description:Improper data protection of Data Index can allow an attacker to read information not intended for disclosure. Review authorization settings.
Justification:Data indexes are stored on the file system and strict file permissions should be enforced to control access to the data store. Built in validations could even be built-in to verify proper file permissions when the process starts.
